# 中文 AIGC 提示词改写（文生图 + 文生视频）

本 Skill 用于把用户输入的中文想法改写成可直接粘贴使用的中文提示词。

**典型用法：** 用户在 **生图 / 生视频之前** 把 **原始中文提示词或一句想法** 贴进对话，希望得到 **可直接复制进模型** 的改写稿（首轮三条差异化 + 每条标明适用模型）；后续可针对某一版 **修订** 或 **换成另一模型的适配版**。

**触发条件：** 见文件顶部 YAML **`description`**（路由关键词、排除场景与边界说明）。

覆盖任务：
- 文生图（text-to-image）
- 文生视频（text-to-video）

覆盖模型族：
- 图像：GPT Image 2.0、Nano Banana
- 视频：Veo 系列、Seedance 系列

---

## 硬性输出规则

### 首轮改写（用户给出原始需求）

1. **必须输出恰好 3 条改写提示词**，供用户挑选；每条必须是完整、可单独复制的正文。
2. 三条之间 **画面/镜头层面的描述要有明确区分度**，不能只在措辞上微调或仅换同义词（见下文「三版本差异化」）。
3. 三条都必须 **忠实用户本意**：同一主体、同一核心动作与叙事目标；差异化只允许落在「表现维度」上（构图、时段光线、景别节奏、氛围侧重等），不得擅自换题材或改剧情走向。
4. **每条标题必须标注适用模型**（短名固定四选一：**GPT Image 2.0**、**Nano Banana**、**Seedance**、**Veo**），与任务类型一致（图任务只用图像模型，视频任务只用视频模型）。输出格式固定为三块：**一行标题（含适用模型）+ 一空行 + 正文**，便于复制：

```
【版本 A · 均衡 · 适用于：Nano Banana】

<完整提示词正文>

【版本 B · 创意 · 适用于：Nano Banana】

<完整提示词正文>

【版本 C · 稳定 · 适用于：Nano Banana】

<完整提示词正文>
```

5. 不在每条前后附加「改写说明」「为什么这样写」；首轮输出 **仅限上述三块**（每块：一行标题 + 正文），不含额外评析段落。
6. 当用户明确指定模型时，三条标题中的「适用于」**须与该模型一致**，三条正文均须使用该模型对应的写法要点。
7. 当用户未指定模型时，自动识别任务类型并选用最匹配模板；**三条默认共用同一路由结果**（同一适用模型）。若用户在首轮就明确 **分版本绑定不同模型**（例如「A/C 用 Nano Banana，B 用 GPT Image 2.0」），允许三条标题中「适用于」不同，但仅限 **同一任务类型内** 的模型组合（图任务只在两种图像模型间分配；视频同理）。未写明分版本绑定时，不按多条模型拆分。
8. 先识别「从零生成」还是「基于参考素材编辑」，并在需要时在每条中体现保留项与改动项。

### 单版本修改（用户指定某一版继续改）

当用户明确表示要针对某一版调整时（例如：「把版本 B 改成黄昏」「在 A 上加雪花」「只改第三条的光」），触发 **修订模式**：

1. **只输出一条**修订后的完整提示词正文（用户指定的那一版的全文替换稿）。
2. 标题格式：`【版本 X · 修订 · 适用于：<模型>】`（X 为 A/B/C；**模型** 与用户上次该版本标注一致，除非用户在本次指令里同时要求更换模型——此时改用目标模型并视为「适配」规则）。
3. 以用户指定版本的上文为准；若对话里没有带出完整旧稿，可简短确认「当前按你描述的版本 X 为准」后输出（仍只占一行标题 + 一空行 + 正文，不写长篇解释）。
4. 修订须满足：保留用户未要求改动部分的语义；仅落实用户提出的修改点；不借机把整个创意换成另一套题材。

### 跨模型适配（用户要「另一模型可用版本」）

当用户明确要求 **换成另一种模型**、**再出一版某某模型**、**「同时给 Nano 和 GPT Image」** 等时，触发 **模型适配**：

1. **输出条数** = 用户要求的适配条数（常见为 1；若用户说「把 A/B/C 分别改成 Veo」则输出 3 条）。
2. 标题格式：`【版本 X · 适配 · 适用于：<目标模型>】`（X 为用户点名的版本；若用户只说「基于上一稿」未点名版本，用 **适配** 且不写版本字母，或写「当前稿」——优先与用户措辞一致）。
3. **正文**：按 **目标模型** 的模板与专属要点 **重写**（结构、占位符、对白写法等切换到位），**保留同一创意内核**（主体、叙事、关键约束），不得偷偷换成另一题材。
4. **任务类型不得擅自跨界**：文生图适配只能在 GPT Image 2.0 ↔ Nano Banana 之间；文生视频只能在 Seedance ↔ Veo 之间。若用户想把「图提示词」改成「视频模型」提示词，须先确认时长与镜头诉求后再输出视频版。
5. 若用户一次性索要 **多个模型** 的多条适配（例如两条：Nano + GPT Image），允许在本轮输出 **多于一条**，每条标题分别标明「适用于」；此类输出 **不属于首轮「恰好三条」**，不受首轮三条限制。

## 三版本差异化（强制）

三条提示词必须在 **至少两类「表现维度」上拉开差距**，例如：

**文生图可组合的维度（每条选不同组合，不必十条全写）：**
- 景别与构图（特写 / 中景 / 远景，平视 / 仰视 / 俯视）
- 时段与光线（清晨 / 正午 / 黄昏 / 夜景；硬光 / 柔光 / 侧光）
- 场景纵深与环境细节（前景道具、背景层次、天气）
- 色彩气质（冷调 / 暖调 / 低饱和电影感）
- 材质与画风写实度（更插画 vs 更纪实）

**文生视频可组合的维度：**
- 镜头序列与切换节奏（一镜到底 vs 明确分镜）
- 机位与运动（固定 / 推轨 / handheld / 航拍感）
- 时间切片内的重心（前段强调环境 vs 后段强调表情）
- 声音侧重（环境音主导 vs 对白主导 vs BGM 气质不同）

**禁止：** 三条只有形容词替换、句式雷同、镜头与场景描述实质相同。

**自检：** 遮住版本标题，随机两条是否在「第一眼画面」上能读出明显不同；若几乎同一画面，必须重写。

## 自动识别流程

按以下顺序判断：

1. 识别任务类型：文生图 / 文生视频
2. 识别模型（若用户已说）
3. 若未说模型，按默认路由：
   - 文生图：优先 Nano Banana；遇到「密集文字排版/UI/品牌字样严格还原」优先 GPT Image 2.0
   - 文生视频：优先 Seedance；遇到「强调对白口型/同步音效/商业旁白质感」优先 Veo

如果任务类型不明确，只问 1 个问题：
- 「你这条是文生图还是文生视频？」

若任务类型明确但关键参数缺失，仅补问 1 个最关键问题后直接输出：
- 图像：比例或用途（二选一）
- 视频：时长（秒）

## 改写原则

- 保留用户核心意图，不擅自改主题。
- 把模糊描述补全为可执行视觉语言：主体、场景、动作、构图、光线、风格、时长/镜头（视频）。
- 避免空泛词堆砌（如「高级感、震撼、大片感」），转为具体可拍/可画细节。
- 对「编辑类请求」采用 delta 改写：明确「改什么」与「保持不变什么」。

## 模板调用

改写前按任务读取对应模板：

- 文生图 → [image-templates.md](references/image-templates.md)
- 文生视频 → [video-templates.md](references/video-templates.md)
- 各模型与官方指南的对照摘要 → [official-sources.md](references/official-sources.md)

不要混用两类模板。

并遵循以下模型专属要点（细则与链接见 `official-sources.md`）：
- **GPT Image 2.0**：顺序贴近官方——**场景/背景 → 主体 → 关键细节 → 约束**；复杂需求可分段或换行；写明 **用途**（电商/UI mock/海报等）；画中字用 **引号** + 字体气质与位置；**编辑**只写「改什么」，并 **反复强调 Preserve**。
- **Nano Banana / Gemini Image**：**正向描述**为主（要什么），否定约束放在句末；结构贴近 **主体 + 动作/情境 + 构图 + 风格**；可按官方示例使用摄影/镜头语言（浅景深、视角等），但 **避免多套互相矛盾的器材参数并列**；画中字同样用引号；编辑明确「改什么 / 保留什么」。
- **Seedance**：**镜头运动**与 **主体表演动作** 分句写清；**单次描述以一个主镜头运动为轴**，避免多条冲突运镜并列；有台词时台词常用 **双引号**；参考素材占位符按接入平台选择：**即梦等国内界面** 常用 `@图片1`，**英文 API 文档** 常用 `[Image1]`/`[Video1]`/`[Audio1]`——用户未说明平台时在三条里统一用一种或简短询问。
- **Veo**：提示词拆解贴近 Vertex「主体、动作、场景、镜头运动、风格氛围」；短片默认 **单场景**；**图生视频** 时侧重描写 **运动与镜头**，少重复静态画面；负面约束优先改写为 **正向「想要的效果」**；对白与音效可参考 DeepMind Veo 3（引号台词 + SFX/环境声分列）。若用户明确走 **Vertex API**，对白可采用「清晰说出以下台词：……」等 **少嵌套引号** 写法以贴合该平台最佳实践文档；若走 **Gemini / DeepMind 示例流**，可保留标准引号对白。有对白时仍须强调 **口型与语音同步**及关键 **环境声/SFX**。

## 输出前质量门（必须全部通过）

1. 首轮是否恰好 3 条；修订模式是否 1 条；**模型适配**是否满足用户要求的条数  
2. **每条标题是否含「适用于：<四模型之一>」** 且与任务类型（图/视频）一致  
3. 首轮：三条是否在画面/镜头维度上有 **可感知的差异**，且核心意图一致  
4. 每条是否一眼可执行（无空话、无解释段落）  
5. 是否覆盖主体/场景/动作/风格/光线（视频还需时长+镜头+节奏）  
6. 是否含清晰约束（避免项、保留项）  
7. 正文写法是否与标题中的 **适用模型** 匹配（适配模式下是否与 **目标模型** 匹配）

## 安全边界

- 不编造用户未要求的敏感身份信息。
- 不添加违法违规、暴力犯罪细节扩写。
- 若用户要求违反平台安全规则，保持中性拒绝并给出可替代、安全版本提示词。
